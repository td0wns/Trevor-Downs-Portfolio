Fresh Start in Education

Salesforce Access Governance, Data Standards and User Adoption

Served as departmental Salesforce super user, governing access to custom objects, fields and Flows while supporting adoption, troubleshooting access issues, and ensuring new architecture reached the correct users without weakening data or process control.

Salesforce & CRM Operations Leadership & Stakeholder Management Data & Analytics
Project overview

Configured and maintained permission sets, object permissions, field-level access and validation rules for standard and custom Salesforce functionality. Tested new architecture across different user-access levels before release, diagnosed access and Flow issues, and supported rollout through documentation, guidance and training so the correct departmental users could adopt new processes without receiving unnecessary permissions.

Key result Created a controlled departmental model for access, data quality and adoption.

01

The governance problem

New Salesforce architecture, custom objects, fields and Flows needed to be available to the correct departmental users without granting unnecessary access or weakening data and process controls. Users also required support, testing, documentation and training to adopt the new functionality successfully.

Object, field, record and Flow access could fail for different reasons, and new architecture needed testing from multiple user-access perspectives before it could be safely released department-wide.

  • Different departmental users needed different levels of access
  • Custom objects and Flows needed to stay usable without over-granting permissions
  • Access issues could originate from objects, fields, records or Flows
  • New architecture required multi-perspective testing before release

02

Permission sets and field-level access

Created and maintained permission sets for users requiring access to specific Salesforce objects, fields and processes, and configured object permissions to control which users could create, read, edit or delete records within standard and custom objects.

Managed field-level access so users could view or update only the information required for their responsibilities, governing access to custom objects including the Attribution Model object and other operational Salesforce architecture built across the wider transformation.

  • Permission sets by user responsibility
  • Object-level create, read, edit and delete controls
  • Field-level access restrictions
  • Governed access to custom objects

03

Guarding Flows and Screen Flows

Ensured intended users could access and complete Salesforce Flows and Screen Flows while preventing inappropriate access, diagnosing and resolving access problems involving objects, fields, records, Flows and related Salesforce functionality as they arose.

Used validation rules alongside permissions to protect controlled processes and prevent incomplete or inappropriate data changes, without blocking the legitimate activity users needed to complete their work.

  • Flow and Screen Flow access configuration
  • Access troubleshooting across objects and records
  • Validation rules alongside permission controls
  • Protection against incomplete or inappropriate changes

04

Testing before departmental release

Tested new Salesforce architecture using different user-access levels before wider departmental release, checking that intended users could complete required processes while users without a legitimate need were correctly restricted.

This multi-perspective testing caught access gaps and overreach before they reached production, reducing the risk of disruption when new objects, fields or Flows went live for the wider department.

  • Multi-perspective access testing before release
  • Verification of intended user journeys
  • Confirmation that restricted users stayed restricted
  • Reduced risk of production disruption

05

Documentation, guidance and training

Supported the rollout of new custom objects, fields, Flows and CRM processes through documentation, guidance and user training, helping users understand how new Salesforce structures affected their daily work rather than leaving adoption to chance.

This translated technical configuration into clear departmental processes, so users could take up new functionality with confidence and without needing repeated one-to-one support for the same issue.

  • User documentation for new functionality
  • Guidance and coaching on changed processes
  • Practical training tied to daily workflows
  • Translation of technical change into plain process

06

Balancing usability and control

Reduced reliance on unrestricted access by assigning permissions according to user responsibilities, balancing user accessibility with data protection, process integrity and operational control across the department.

This work covered departmental super-user support, configuration and adoption responsibility, distinct from organisation-wide Salesforce platform ownership, and gave the organisation a repeatable model for rolling out future Salesforce changes safely.

  • Permissions aligned to user responsibility
  • Usability balanced against data protection
  • Clear departmental (not org-wide) ownership scope
  • Repeatable model for future safe rollouts

Outcomes

What the access and governance model enabled

Right access, right people

Permission sets, object permissions and field-level security aligned access with actual user responsibilities rather than blanket permissions.

Protected custom architecture

Governed access to custom objects and Flows, including the Attribution Model object, so intended users could work without unnecessary exposure.

Fewer surprises at release

Multi-perspective testing before rollout reduced access-related disruption when new functionality went live.

Faster, smoother adoption

Documentation, guidance and training turned technical configuration into clear departmental process, reducing repeated one-to-one support.
